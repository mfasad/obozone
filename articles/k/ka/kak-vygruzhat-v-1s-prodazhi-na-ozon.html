Как выгружать продажи с Озон в 1С: полное руководство для продавцов

Автоматизация учета продаж с маркетплейсов — ключевой фактор роста для любого продавца на Ozon. Ручное внесение заказов в 1С:Предприятие отнимает часы работы, увеличивает риск ошибок в отчетности и замедляет обработку заказов. По данным исследования Data Insight за 2023 год, продавцы, использующие автоматическую выгрузку данных с маркетплейсов, сокращают время на обработку заказа на 40% и снижают количество ошибок в учете на 78%.

Но как правильно настроить этот процесс? Какие инструменты выбрать — API Ozon, EDI-обмен или выгрузку через файлы? И какие подводные камни ждут новичков? В этой статье разберем все этапы: от подготовки 1С до обработки типичных ошибок синхронизации.

Если вы только начинаете работать с Ozon или уже столкнулись с проблемами при выгрузке данных, эта инструкция поможет настроить процесс с нуля. Мы рассмотрим актуальные методы интеграции на 2026 год, включая изменения в API Ozon Seller и новые требования к формату данных. Особое внимание уделим обязательной настройке статусов заказов в 1С для корректной работы с FBS/FBO — этот момент упускают 6 из 10 продавцов, что приводит к штрафам за несвоевременную отгрузку.

Важно: статья ориентирована на пользователей 1С:Управление торговлей (ред. 11.4 и выше), 1С:ERP и 1С:Комплексная автоматизация. Для других конфигураций (например, 1С:Бухгалтерия) потребуются доработки или дополнительные модули.

1. Подготовка 1С к выгрузке данных с Озон

Прежде чем настраивать обмен, убедитесь, что ваша 1С готова к работе с данными маркетплейса. Без правильной подготовки даже идеально настроенный обмен приведет к дублям заказов или потере информации.

Первый шаг — обновление конфигурации. Для работы с Ozon требуется версия платформы не ниже 8.3.18 (для API-интеграции) или 8.3.15 (для EDI). Проверить версию можно в меню Справка → О программе. Если ваша версия устарела, обновите ее через Конфигуратор или обратитесь к партнеру 1С.

Второй критичный момент — настройка справочников. В 1С должны быть заведены:

  • 📦 Контрагенты: Ozon как покупатель (для FBS) и как партнер (для FBO)
  • 🏷️ Номенклатура: все товары с артикулами, совпадающими с SKU на Озон
  • 📍 Склады: отдельные склады для FBS и FBO (если используете обе схемы)
  • 💰 Типы цен: цена продажи на Озон (может отличаться от розничной)

Третий этап — создание пользователя для обмена. В 1С должен быть отдельный пользователь с правами на чтение/запись заказов, номенклатуры и остатков. Не используйте для интеграции учетную запись администратора — это создает риски безопасности.

⚠️ Внимание: Если вы работаете по схеме FBO, в 1С отдельный вид операции"Реализация товаров (комиссия)" с указанием Ozon как комитента. Без этого выгрузка заказов будет приводить к искажению финансового результата.

2. Методы выгрузки продаж: какой выбрать?

На 2026 год у продавцов Ozon есть три основных способа выгрузки заказов в 1С. Каждый имеет свои плюсы и минусы — выбор зависит от объема продаж, бюджета и технических возможностей.

Рассмотрим все варианты в сравнительной таблице:

API Ozon Seller — самый современный и гибкий способ. Он позволяет получать заказы в реальном времени, обновлять статусы автоматически и даже управлять ценами. Однако для его использования потребуется:

  • 🔑 Client-ID и API-key (получаются в личном кабинете Ozon Seller)
  • 🖥️ Доступ к серверу 1С для настройки HTTP-соединений
  • 📜 Знание основ JSON и OAuth 2.0 (или готовое решение от партнеров)

Выгрузка через файлы — самый простой, но и самый трудоемкий способ. Подходит для тестирования или малых объемов продаж. Ozon позволяет скачивать отчеты по заказам в формате XLS или CSV, которые затем можно загрузить в 1С через стандартную обработку Загрузка данных из табличного документа.

3. Пошаговая настройка выгрузки через API Ozon

Если вы выбрали API-интеграцию, следуйте этой инструкции. Весь процесс займет от 2 до 8 часов в зависимости от опыта работы с 1С и API.

Шаг 1: Получите доступ к API Ozon

  1. Авторизуйтесь в личном кабинете Ozon Seller
  2. Перейдите в раздел Настройки → Интеграции → API-ключи
  3. Создайте новый ключ с правами Чтение и запись заказов, Чтение информации о товарах
  4. Скопируйте Client-ID и API-key — они понадобятся для настройки в 1С

Шаг 2: Настройте HTTP-соединение в 1С

В конфигураторе 1С выполните:

  1. Откройте Администрирование → HTTP-сервисы
  2. Создайте новый веб-сервис с адресом https://api-seller.ozon.ru
  3. Укажите метод аутентификации OAuth 2.0 и введите полученные Client-ID/API-key
  4. Настройте обработчик для приема данных в формате JSON

Шаг 3: Создайте обработку для обмена данными

Здесь есть два пути:

  • 🛠️ Разработать собственную обработку (требует знания 1С:Предприятие и REST API)
  • 📦 Использовать готовые решения:
    • 1С-Битрикс: Интеграция с Ozon (от 15 000 руб.)
    • Модуль"Ozon + 1С" от"АйТи-Проект" (от 20 000 руб.)
    • Облачный сервис"МойСклад" или"Эвторика" (от 3 000 руб./мес)

Шаг 4: Настройте расписание обмена

Для автоматического обновления заказов каждый час добавьте задание в Планировщик 1С:

ОбменДаннымиСOzon.ВыполнитьОбновлениеЗаказов;

ОбменДаннымиСOzon.ОбновитьСтатусыОтгрузок;

⚠️ Внимание: При настройке API обратите внимание на лимит запросов — Ozon позволяет делать не более 100 запросов в минуту на один Client-ID. При превышении лимита API блокируется на 1 час. Чтобы избежать этого, настройте паузу между запросами в 1-2 секунды.
💡

Перед запуском API-интеграции в боевом режиме протестируйте обмен на копии базы 1С. Для этого создайте тестового пользователя в Ozon Seller с правами только на чтение заказов за последний день.

4. Особенности выгрузки для схем FBS и FBO

Процесс выгрузки заказов существенно отличается в зависимости от того, по какой схеме вы работаете: FBS (доставка силами Ozon) или FBO (самостоятельная доставка). Ошибки в настройке приводят к тому, что заказы либо не выгружаются, либо попадают в неправильные статусы.

Для FBS критично правильно настроить:

  • 📦 Склад отгрузки: в 1С должен быть указан склад Ozon FBS (не ваш физический склад!)
  • 🚚 Статусы заказов: заказы должны автоматически переходить в статус"Отгружен" при получении уведомления от Ozon о передаче товара на склад
  • 💰 Комиссию маркетплейса: в 1С должна быть настроена статья затрат"Комиссия Ozon FBS" (обычно 15-17%)

Для FBO важно:

  • 🏠 Склад отгрузки: ваш реальный склад, с которого будет отправляться товар
  • 📋 Печать этикеток: в 1С должна быть обработка для генерации транспортных этикеток Ozon в формате PDF или ZPL
  • 📦 Упаковка: если используете Ozon Box, в заказе должен проставляться соответствующий признак

Рассмотрим на примере, как отличается обработка одного и того же заказа в двух схемах:

Действие FBS FBO
Склад в заказе 1С Ozon FBS (виртуальный) Основной склад компании
Статус после оплаты "Ожидает отгрузки на FBS" "Оплачен, готов к сборке"
Ответственный за доставку Ozon (автоматически) Ваша логистика
Документ реализации Акт передачи товара на FBS Реализация товара покупателю

Один из самых распространенных косяков — неправильная настройка статусов. Например, когда заказ FBS попадает в статус"Готов к отгрузке", хотя должен быть"Передан на склад Ozon". Это приводит к тому, что:

  • 📉 В отчетности появляются"висящие" заказы
  • 💸 Ozon штрафует за несвоевременную отгрузку (до 1 000 руб. за заказ)
  • 📦 Товар может быть резервирован дважды — и на вашем складе, и на FBS
Как проверить правильность настройки статусов?

Создайте тестовый заказ на Ozon (можно через личный кабинет в режиме"Песочница").

Отследите, как он проходит по статусам в 1С:

1."Оформлен" → должен появиться в 1С как"Новый заказ"

2."Оплачен" → должен перейти в"Ожидает отгрузки" (FBS) или"Готов к сборке" (FBO)

3."Отгружен" → должен закрыться документом реализации с правильной датой

Если на любом этапе статус не совпадает — проверьте настройки обмена в модуле интеграции.

5. Типичные ошибки и их решения

Даже при правильной настройке интеграции продавцы сталкиваются с ошибками. Вот самые распространенные проблемы и способы их решения:

Ошибка 1:"Не найден артикул товара"

  • Причина: Расхождение SKU в 1С и на Ozon
  • Решение:
    1. Экспортируйте список товаров из Ozon Seller (Товары → Экспорт)
    2. Сравните артикулы с данными в 1С (справочник"Номенклатура")
    3. Приведите к единому формату (уберите пробелы, приведите к верхнему регистру)

Ошибка 2:"Некорректный статус заказа"

  • Причина: В 1С не настроено соответствие статусов Ozon и внутренних статусов
  • Решение:
    1. Откройте обработку обмена данными
    2. Найдите таблицу соответствия статусов (обычно в разделе"Настройки")
    3. Добавьте недостающие статусы (например,"awaiting_packaging" →"Ожидает упаковки")

Ошибка 3:"Превышен лимит API-запросов"

  • Причина: Слишком частые запросы к API (больше 100 в минуту)
  • Решение:
    1. Настройте паузу между запросами в обработке (рекомендуется 2 секунды)
    2. Используйте кэширование данных (сохраняйте ответы API в регистры сведений)
    3. Разбейте большие пакеты запросов на более мелкие

Ошибка 4:"Не совпадают суммы заказа"

  • Причина: В 1С не учтена комиссия Ozon или скидки
  • Решение:
    1. Проверьте настройку статьи затрат"Комиссия Ozon" (обычно 15-17%)
    2. Убедитесь, что в заказе учитываются промокоды и скидки маркетплейса
    3. Сверьте настройки округления сумм в 1С и на Ozon

    • Если ошибка нечная, проверьте логи обмена в 1С:

    1. Откройте Администрирование → Журналы регистрации
    2. Выберите период, когда произошла ошибка
    3. Найдите записи с типом"ОбменДаннымиСOzon"
    4. Посмотрите детали ошибки в колонке"Сообщение"
    ⚠️ Внимание: Если в логах встречается ошибка 429 Too Many Requests, это означает, что ваш API-key временно заблокирован. Подождите 1 час и уменьшите частоту запросов. Повторная блокировка в течение суток может привести к полному отключению API на 24 часа.
    💡

    90% ошибок при выгрузке заказов связаны с тремя проблемами: несовпадение артикулов, неправильные статусы или лимиты API. Всегда начинайте диагностику с проверки этих пунктов.

    6. Автоматизация дополнительных процессов

    Выгрузка заказов — только часть задачи. Для полной автоматизации учета настройте еще три критичных процесса:

    1. Обновление остатков на Ozon

    • 🔄 Настройте автоматическую выгрузку остатков из 1С на Ozon 2-3 раза в день
    • 📊 Используйте обработку, которая учитывает:
      • Резерв под заказы (чтобы не продавать товар в двойную)
      • Минимальный остаток (например, не выгружать остатки меньше 3 шт.)
      • Задержку поставки (если товар в пути)

    2. Синхронизация цен

    • 💰 Настройте правила ценообразования:
      • Базовая цена + наценка/скидка для Ozon
      • Автоматическое округление до 9 или 99 рублей
      • Учет комиссии Ozon (например, цена в 1С = цена на Ozon × 1.17)
    • ⏱️ Обновляйте цены не чаще 1 раза в 2 часа (лимит Ozon — 10 обновлений цен в день на один товар)

    3. Обработка возвратов

    • 🔙 Настройте автоматическое создание документов"Возврат товара" в 1С при получении уведомления от Ozon
    • 📦 Учитывайте разные типы возвратов:
      • Возврат на склад Ozon (для FBS)
      • Возврат на ваш склад (для FBO)
      • Частичный возврат (когда возвращают не весь заказ)

    Для автоматизации этих процессов можно использовать:

    • 🛠️ Стандартные механизмы 1С: обработки"Обмен с сайтом" и"Загрузка данных"
    • 📦 Готовые модули:
      • 1С:Интеграция с маркетплейсами (от 25 000 руб.)
      • Атол: Обмен с Ozon (от 18 000 руб.)
      • Клеверенс: Маркетплейсы (от 12 000 руб./мес)
    • ☁️ Облачные сервисы:
      • МойСклад (интеграция с 1С и Ozon)

    • 🧪 Песочница Ozon: тестовый режим в личном кабинете, где можно создавать фиктивные заказы
    • 📊 Журнал регистрации 1С: отслеживайте все действия системы во время теста
    • 🔍 Postman: для ручного тестирования API-запросов перед настройкой в 1С

    Если тестирование прошло успешно, можно переходить к боевому запуску. Начните с постепенного включения:

    1. Первые 2-3 дня выгружайте только новые заказы (без исторических данных)
    2. Следите за журналом обмена на предмет ошибок
    3. Сверяйте количество заказов в 1С и личном кабинете Ozon раз в час
    4. Постепенно подключайте дополнительные функции (обновление цен, остатков)

    Типичные проблемы при запуске:

    • 🐢 Медленная выгрузка: обычно связана с большим объемом исторических данных. Решение — выгружать данные порциями (например, заказы за последние 3 дня, затем за неделю и т.д.)
    • 🔄 Дублирование заказов: проверьте настройки уникальных идентификаторов в обработке обмена
    • 💻 Зависание 1С: увеличьте лимит памяти для 1С:Предприятия (в файле 1CEStart.cfg добавьте строку /MaxMemory 4096)

    8. Оптимизация и масштабирование

    Когда интеграция работает стабильно, можно переходить к оптимизации процессов. Вот несколько советов для увеличения скорости и надежности обмена:

    1. Оптимизация запросов к API

    • 📡 Используйте пакетные запросы: вместо отдельных запросов для каждого заказа получайте данные пачками по 50-100 штук
    • 🗃️ Настройте кэширование: сохраняйте ответы API в регистры сведений, чтобы не запрашивать одни и те же данные повторно
    • ⏱️ Уменьшите частоту опроса: для большинства магазинов достаточно обновлять заказы раз в 30-60 минут

    2. Автоматизация рутинных операций

    • 📑 Настройте автоматическую печать документов:
      • Счета-фактуры для FBO
      • Транспортные этикетки для FBS
      • Упаковочные листы
    • 📧 Подключите email/SMS-уведомления:
      • Оповещение менеджера при новых заказах
      • Уведомление логиста при изменении статуса на"Готов к отгрузке"
      • Предупреждение бухгалтеру о новых возвратах

    3. Мониторинг и аналитика

    • 📈 Настройте дашборды в 1С или Power BI для отслеживания:
      • Динамики продаж по дням/неделям
      • Среднего чека и конверсии
      • Количества возвратов и их причин
    • 🛠️ Используйте систему оповещений о сбоях:
      • Уведомление при ошибках API
      • Предупреждение о низких остатках популярных товаров
      • Сигнал о превышении лимитов API-запросов

    Для масштабирования можно рассмотреть:

    • ☁️ Переход на облачную 1С: если локальный сервер не справляется с нагрузкой
    • 🤖 Подключение RPA-роботов: для автоматизации рутинных операций (например, обработка возвратов)
    • 📊 Интеграция с WMS: если работаете с большими складами (от 5 000 SKU)
